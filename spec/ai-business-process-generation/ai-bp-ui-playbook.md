## 1. Designer technology (what we're automating)

- The classic Process Designer (`…/0/Nui/ViewModule.aspx?vm=SchemaDesigner#process/<id>`)
  is now built on **diagram-js / bpmn-js (bpmn.io)**, bundled as the
  `app_studio_enterprise_process_designer` webpack chunk, and **embedded inside the
  legacy Terrasoft/Ext "classic" shell**.
- The host Ext control is `Terrasoft.controls.ProcessDiagramComponent` (DOM id
  `schema-designer`). It hosts the Angular/diagram-js sub-app and bridges it to a
  Terrasoft view model via `Terrasoft.diagram.DiagramChangesParser`.
- Canvas = **SVG**. Each element is a diagram-js shape; its **type is encoded in the
  CSS class**, e.g. `djs-element djs-shape readDataUserTask`, `… startEventSignal`,
  `… formulaTask`, `… endEvent`, `… callActivity`; connections are
  `djs-element djs-connection`. Element ids are GUIDs in `data-element-id`.
- A brand-new process (`#process/` with no id) bootstraps as "Business process N" with
  a default **Start → End** already on the canvas.

### Render caveat (important for any harness)
The SVG canvas + left palette render **a few seconds AFTER** the right-hand metadata
card. A screenshot taken too early shows a blank/!black canvas. Poll for
`document.querySelector('.djs-shape')` (or `.entry[data-action="create-serviceTask"]`)
before interacting. A stale browser cache can also break the first render (cleared during recon).

## 4. Automation feasibility findings (the crux for the channel decision)

Tested empirically during recon:

1. **Untrusted JS events do NOT work.** Dispatching `MouseEvent`/`PointerEvent` via
   `evaluate_script` (`isTrusted=false`) does **not** open the create-menu (it stays
   `visibility:hidden`) nor drive diagram-js drag-to-place. The designer relies on trusted
   input + pointer capture.
2. **uid-based MCP click works for menu items, but canvas placement is blocked.** The
   System/User-action menu items appear in the a11y snapshot (addressable by the
   chrome-devtools `click` tool, which uses trusted CDP input). BUT the SVG canvas drop
   target has **no uid**, and the MCP `drag` tool requires *both* endpoints as uids → there
   is no way to drop an element at an arbitrary canvas coordinate with the current MCP toolset.
3. **Robust paths identified:**
   - **(A) Trusted coordinate input** — Playwright or raw CDP `Input.dispatchMouseEvent`
     (or the Claude-in-Chrome `computer` tool) can drag palette→canvas coordinates and fill
     cards. This is the genuine "drive it like a human" Approach-3 path; needs a real harness,
     not the uid-based MCP click/drag.
   - **(B) Injected-JS against the designer's own API** — build the process as **schema items**
     via `Terrasoft.addSchemaItems` / `insertSchemaItemsAfter` / `applySchemaItemProperties`
     (+ `Terrasoft.ProcessSchemaElementsInitializer`) on the `ProcessDiagramComponent` model,
     letting diagram-js re-render. Most reliable at runtime; couples to internal APIs
     (version-fragile) — effectively a *client-side* version of research Approach 1.
   - **(C) Context-pad "append"** (untested) — hovering an existing element may expose an
     "Append" affordance that auto-places the next element (no coordinate drop needed); if it
     is trusted-clickable + snapshot-addressable, pure chrome-devtools MCP could suffice.

### Implication for the plan's channel decision
"Pure human-like puppeteering via chrome-devtools MCP" is **insufficient on its own** for the
canvas interactions (confirmed). The PoC needs **(A)** a trusted-coordinate harness, or **(B)**
the injected-API path, or to validate **(C)** the append affordance. The element catalog (§2)
is channel-independent and carries straight into subtask 1 and research Approach 1.

## 6. PROVEN PoC recipe (validated end-to-end 2026-06-12)

Built **Start → Read data (Contact) → End**, saved it, and verified it via clio —
all through chrome-devtools MCP. The winning combination is **trusted MCP clicks for
menus/cards + a tiny untrusted drag for canvas append**, exactly as the QA framework
(`angular_process_designer_page.js`) does it.

### Step-by-step (what actually worked)

1. **Auth + open**: navigate `/Login/NuiLogin.aspx`, fill Supervisor/Supervisor, LOG IN →
   navigate `…?vm=SchemaDesigner#process/`. Poll for `.djs-shape` before acting (SVG renders
   a few seconds after the metadata card; clear cache if the canvas stays black).
   A fresh process starts as `Start → End`.
2. **Select a shape (canvas has no uid → overlay trick)**: read the shape's screen rect via
   `getBoundingClientRect`, inject a `position:fixed; pointer-events:none` `<div>` with visible
   text over it, snapshot to get the overlay uid, then the trusted MCP `click` lands at those
   coords and passes through to the diagram-js hit layer (`rect.djs-hit.djs-hit-all`).
   ⚠️ Close any stray `.djs-popup.diagram-create-popup-menu` first (Escape) — it floats over the
   canvas and intercepts the click.
3. **Append the element (the one untrusted bit, QA-proven)**: with the source selected, its
   context pad shows `add.userTask` / **`add.serviceTask`** / `add.exclusiveGateway` /
   `add.endEvent` / `connect` / `suggestions`. Dispatch (untrusted) on
   `.djs-context-pad .entry[data-action="add.serviceTask"]`: `dragstart` → `mousemove` over the
   source → grab the last `.djs-drag-group` → `mousemove`/`mouseover`/`mousemove`/`mouseup` at the
   target canvas point. **`add.serviceTask` defaults to `readDataUserTask` (Read data)** and
   auto-inserts onto the Start→End flow → valid `Start → Read data → End`. (For other types: drop
   the default, then morph via `[data-action=setup]` → `[data-id="ServiceTask"|"UserTask"]` →
   `[data-id="<type>"]` — all trusted clicks, no drag.)
4. **Configure the setup card (right panel — classic DOM, fully snapshot-addressable)**: the card
   opened automatically. Fields seen for Read data: "Which data read mode to use?"
   (= "Read the first record in the selection"), **"Which object to read data from?"** (lookup).
   Fill the lookup with `Contact` → its `.listview` dropdown appears → trusted-click the "Contact"
   option. The card then expands with "How to filter records?" (filter builder), "How to sort
   records?", "What record data should the process read?" (= all columns), plus an **AI SUGGESTIONS**
   panel proposing next elements.
5. **Title + Save**: set the caption textbox; click **SAVE** → message-panel **"Successfully saved"**
   (no validation dialog was needed for this minimal valid process).

### Verification (clio, env `krestov-test` = the same instance)
- `execute-esq` on `VwProcessLib` filtered by Caption → 1 row: Name `UsrProcess_493d4c9`,
  Caption "AI PoC Read Contact", type "Business process", UId `dd3a473e-…`.
- `generate-process-model --code UsrProcess_493d4c9` → exit 0, emitted

### Channel verdict (resolves the plan's open decision)
Pure **chrome-devtools MCP is sufficient** for this flow after all — the canvas-addressing wall is
solved by the `pointer-events:none` overlay + trusted click, and the only untrusted step (context-pad
append drag) is the same one the QA suite relies on. A Playwright harness (Path A) remains the
productization option for headless/CI, reusing this exact recipe and `get-browser-session` auth.
The diagram-js internals API (Path B) is the fallback for steps that resist trusted input.

## 7. Live tuning of the driver recipe (krestov-test, 2026-06-12) — confirmed selectors

Running `process-add-element` live (auth+launch+navigate succeeded — "injected 8 session cookie(s)")
surfaced three recipe fixes, each then validated phase-by-phase against the live designer:

1. **append** — the diagram-js `.djs-drag-group` only materializes **after an initial `mousemove` on the
   source element**. Sequence: `dragstart` on `.djs-context-pad .entry[data-action="add.serviceTask"]`
   → `mousemove` on the Start shape (center) → grab the last `.djs-drag-group` → `mousemove`/`mouseover`/
   `mousemove`/`mouseup` at the canvas target. (Symptom before the fix: "append drag did not start".)
2. **object lookup** — the "Which object to read data from?" field is `input[id*="EntitySchemaSelect"]`
   (full id `ReadDataUserTaskPropertiesPageEntitySchemaSelectComboBoxEdit-el`). Untrusted value-set +
   `input`/`keyup` opens the `.listview` dropdown; click the option whose text === the object (trusted).
   (Related ids: read mode `...DataReadModeComboBoxEdit-el`; element caption `...captionTextEdit-el`.)
3. **SAVE** — the toolbar button is `#schema-designer-save-btn-textEl` (DOM text "Save", CSS uppercases
   to "SAVE" — an exact `=== 'SAVE'` match misses it). Process caption field: `#schema-designer-caption-el`.

Save signal: a `.message-panel` containing "Successfully saved" (confirmed). `SchemaUId` read-back from
`Ext.ComponentManager.get('schema-designer').model.values` returned null — identity is resolved instead by
caption via `describe-process` / `execute-esq` (the OQ-04 readback handle).

**Live end-to-end result:** built Start → Read data(Contact) → End, saved, and read it back with the clio
`describe-process` MCP tool → `UsrProcess_6ddeb29` ("AI live e2e read contact tuned"), elements
StartEvent/ReadDataUserTask1/TerminateEvent + two sequence flows. Note: `describe-process` types the Read
data element as `dataId:"Unknown"/type:"Other"` because the data-operation element's `managerItemUId` is not
in `ManagerMap`'s GUID map — a future refinement (map the data-task manager UIds for richer read-side typing);
the element is still read with its label + params.
