1. **Анализ и чтение файла настроек**
	- Проверить наличие файла `.clio/workspacesettings.json` в корне workspace.
	- Реализовать парсер для чтения ключа `ignorePackages` (массив строк).
	- Поддержать маски (wildcards) и относительные пути в значениях.

2. **Интеграция в команды работы с workspace**
	- Внести изменения в команды, которые обрабатывают пакеты workspace (`pushw`, `restorew`, публикация, сборка и т.д.).
	- Перед обработкой пакетов фильтровать их по списку из `ignorePackages`.
	- Убедиться, что игнорируемые пакеты не удаляются и не изменяются командами.

3. **Поддержка масок и путей**
	- Реализовать сопоставление имён пакетов с масками (`*Test*`, `Demo*` и т.д.).
	- Использовать существующую файловую абстракцию для поиска и фильтрации.

4. **Тестирование**
	- Добавить unit-тесты для:
	  - Чтения и парсинга файла настроек.
	  - Фильтрации пакетов по маскам.
	  - Проверки корректной работы команд с игнорируемыми пакетами.
	- Применить эквивалентные классы, boundary value analysis и pairwise testing.

5. **Документация**
	- Обновить `README.md` и `Commands.md`:
	  - Описать новый механизм игнорирования.
	  - Привести примеры использования и формат файла.
	  - Указать ограничения и рекомендации.

6. **Обработка ошибок**
	- Реализовать обработку ошибок при отсутствии файла, некорректном формате, пустом списке и т.д.
	- Добавить сообщения для пользователя о том, какие пакеты были проигнорированы.
## Задание: Игнорирование пакетов в workspace через настройки

**Цель:**
Реализовать механизм игнорирования пакетов при операциях с workspace (например, публикация, сборка, экспорт и т.д.)

**Требования:**
- Список игнорируемых пакетов должен храниться в файле `.clio/workspacesettings.json` в корне workspace.
- Формат файла:
	```json
	{
		"ignorePackages": [
			"PackageName1",
			"PackageName2"
		]
	}
	```
- При выполнении операций с workspace, пакеты из списка `ignorePackages` должны быть полностью исключены из обработки.
- Если файл или ключ отсутствует, обработка всех пакетов стандартная.
- Должна быть поддержка относительных путей и масок (например, `*Test*`, `Demo*`).
- Все изменения должны быть покрыты unit-тестами.
- Документация по использованию должна быть добавлена в README.md и Commands.md.

**Пример:**
В файле `.clio/workspacesettings.json`:
```json
{
	"ignorePackages": [
		"TestPackage",
		"Demo*"
	]
}
```
При публикации workspace пакеты `TestPackage` и все, начинающиеся с `Demo`, игнорируются.
