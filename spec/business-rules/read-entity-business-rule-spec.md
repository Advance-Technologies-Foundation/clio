# read-entity-business-rule

## Purpose

This capability defines behavior specific to **reading** entity-level business rules. It is the
reverse of [create-entity-business-rule-spec.md](./create-entity-business-rule-spec.md): given a
target entity, it returns every business rule configured on it, reconstructed into the same friendly
contract that create consumes.

Design rationale and trade-offs: [read-business-rules-adr.md](./read-business-rules-adr.md).

## Consumer

The capability is designed for one consumer:

- an AI agent (or a human via the CLI) that **inspects** the rules already on an entity — to
  understand current behavior or to avoid creating a duplicate — and optionally **recreates** one
  elsewhere by feeding the returned item straight back into
  [create](./create-entity-business-rule-spec.md) with no field-stripping.

Editing and deleting existing rules are **out of scope**. Because the inspect/recreate consumer needs
exactly `caption` / `condition` / `actions`, the output is the byte-identical create contract — no
identity (`name`/`uId`) and no `enabled` flag. A future edit/delete capability will introduce a rule
identity handle when it has a real caller.

> Deliberate deviation from the earlier [ENG-88580-plan.md](./ENG-88580-plan.md), which proposed a
> separate `list-entity-business-rules` summary tool plus a `get-entity-business-rule` that surfaced
> `name` and `enabled`. This capability is a single get-all tool returning the pure create contract.

## Behavior

Entity-level rule reading must:

- accept a target environment, package, and entity schema name; require no rule body on input
- load current business-rule metadata from `AddonSchemaDesignerService.svc` for the resolved package
  and entity schema (the same add-on channel create writes to)
- return ALL rules for the entity as an array, **preserving stored metadata order**
- return each rule as the friendly create contract `{ caption, condition, actions }` and nothing else
- resolve each rule's `caption` from its `{uId}.Caption` add-on resource, falling back to the caption
  stored on the rule metadata when the resource is absent
- reconstruct conditions as the mirror of create:
   - condition-group `logicalOperation` from its integer code (1 → `AND`, 2 → `OR`)
   - each condition's `comparisonType` from its integer code (inverse of the create mapping)
   - unary comparisons (`is-filled-in` / `is-not-filled-in`) omit `rightExpression`; binary
     comparisons include it
   - left/right expressions of every supported type: `AttributeValue` (path), `Const` (value typed by
     its stored `dataValueTypeName`), `SysValue` (`sysValueName`), and `Formula`
- reconstruct actions as the mirror of create:
   - `make-editable` / `make-read-only` / `make-required` / `make-optional` from the stored
     element/attribute list
   - `set-values` items as constant, attribute, or **formula** values; the friendly formula text is
     recovered from the stored expression by stripping the `#{EntitySchema}Record.<path>#` wrapper
   - `apply-filter` from the parent lookup-filter action: `target`, `targetFilterPath`, `source`,
     `sourceFilterPath`, `clearValue`, `populateValue`
   - `apply-static-filter` by decompiling the stored ESQ envelope back into the friendly filter
     contract (`logicalOperation`, `filters`, `groups`, `backwardReferenceFilters`), including:
      - leaf comparisons and unary `is-null` / `is-not-null`
      - lookup `IN` filters, with display values recovered from the names embedded in the envelope
        (no live lookup queries)
      - relative-date / current-user macros (with their integer argument where applicable)
      - `datePart` extractions (calendar/clock parts and time-of-day)
      - nested groups
      - backward-reference `EXISTS` / `NOT_EXISTS` and `COUNT` / `SUM` / `AVG` / `MIN` / `MAX`
        aggregations
- **collapse `apply-filter` child rules:** the auto-generated `Autogenerated_{parentUId}_ClearValue`
  / `_PopulateValue` rules (linked by `parentUId` / `parentActionUId`) are excluded from the array and
  folded back into the single parent `apply-filter` action
- ignore trigger metadata (it is derived on create and is not part of the friendly contract)
- return disabled rules in their create-contract form like any other rule (the `enabled` state is not
  surfaced, so a disabled rule is indistinguishable from an enabled one)

## Correctness bar

The read result is **round-trip stable**: feeding it back into create reproduces the same stored
metadata (read → create → read is stable / re-compiles to the same ESQ envelope). Output is not
guaranteed byte-identical to the original human input — value formatting (e.g. float/datetime) may
be normalized. Round-trip property tests over the create-side filter fixtures enforce this bar.

## Validation / failure

Entity-level rule reading must fail with a clear error when:

- the entity schema cannot be resolved
- the package cannot be resolved
- the add-on metadata cannot be loaded or is not valid JSON
- a stored rule cannot be faithfully reconstructed into the friendly contract — the error **names the
  offending rule**; the read does not return raw or partial metadata for unsupported shapes
