# ADR: Read entity business rules via a metadata reverse converter + ESQ decompiler

- Status: Accepted
- Date: 2026-06-24
- Feature: read-business-rules (entity scope, first cut)
- Supersedes: an earlier proposed ADR (now removed) that assumed a create-time
  `{uId}.ClioFriendlySource` persistence approach
- Related: `create-entity-business-rule-spec.md`, `business-rules-architecture.md`,
  ENG-88580 (earlier list+get sketch), PR #732 / ENG-91313 (batch create)

## Context

Business-rule **creation** is a one-way compile: the friendly contract
(`caption` / `condition` / `actions`) is converted by `BusinessRuleMetadataConverter` into verbose
Creatio add-on metadata and persisted via `AddonSchemaDesignerService.svc`
(`BusinessRuleAddonService`). No reverse path exists; the only code that reads rules back is the
E2E assertion helper `clio.mcp.e2e/Support/Creatio/BusinessRuleAddonReadback.cs`, which navigates
raw JSON for assertions and never reconstructs a friendly rule.

We want a **read** capability so a coding agent (or a human via the CLI) can inspect the rules
already configured on an entity and optionally recreate one elsewhere. This requires building the
missing persistence → friendly reverse direction.

Investigation of the forward path established that the reverse is bounded and self-contained:

- **Conditions, `make-*`, `hide`/`show`, const/attribute `set-values`, and `apply-filter`** reverse
  mechanically from stored metadata — a clean mirror of `BusinessRuleMetadataConverter`.
- **`set-values` formula is *not* lossy.** `BusinessRuleFormulaBuilder` performs a transparent token
  rewrite — `(Field1 + Field2) / 2` → `(#ContactRecord.Field1# + #ContactRecord.Field2#) / 2` — and
  retains the text in `expressionSchema.Expression`. The friendly formula is recovered by stripping
  the deterministic `#{EntitySchema}Record.<path>#` wrapper.
- **`apply-static-filter` is the only hard case.** The friendly filter compiles to an opaque ESQ
  envelope JSON via `LocalEsqFilterBuilder` (≈465 LOC). **Key de-risking finding:** the forward
  builder *embeds the lookup display name in the envelope* (`EsqLookupValueDto.Name` /
  `DisplayValue`, set mandatorily at `LocalEsqFilterBuilder.cs:152-170`). Therefore GUID→display is
  recoverable **from the stored metadata alone — no live Creatio queries are required.** The wire
  format is fully described by `Filters/Esq/EsqEnvelopeDtos.cs` + `EsqFilterEnums.cs`, so a
  deterministic inverse is feasible.

## Decision

1. **Surface.** Add one read-all tool, **`get-entity-business-rules`**, that returns ALL business
   rules for a target entity as a `rules[]` array in stored metadata order. (Page read is a
   documented follow-up. The single-array shape supersedes the earlier ENG-88580 list+get sketch.)

2. **Output contract — byte-identical to create (Option 1).** Each array item is exactly the create
   contract `{ caption, condition, actions }`. **No `name`, no `enabled`, no `uId`.** The response is
   just the array (no per-item created/failed envelope — that belongs to write operations).

3. **Defined consumer.** An agent or human that **inspects** existing rules (understand behavior /
   avoid duplicates) and optionally **recreates** one by feeding the item straight back into
   `create-entity-business-rule(s)` with zero field-stripping. Edit/delete are **out of scope**;
   they will introduce a rule-identity handle (`name`) when those tools exist and a real caller
   needs it — we do not ship identity fields with no consumer now (YAGNI).

4. **Full ESQ decompiler, no create-path change.** `apply-static-filter` is read by a new
   `EsqFilterDecompiler` — the deterministic inverse of `LocalEsqFilterBuilder` — which reconstructs
   the friendly filter from the stored envelope (lookup display names read from the embedded
   `EsqLookupValueDto.Name`). The create path is **not** modified. Lossless for every rule, including
   UI-authored ones.

5. **`apply-filter` collapse.** The parent rule's `BusinessRuleFilterLookupActionMetadataDto` carries
   `ClearValue` / `PopulateValue` and both lookup paths; the auto-generated child rules
   (`Autogenerated_{parentUId}_ClearValue/_PopulateValue`, linked by `parentUId` / `parentActionUId`)
   are detected, excluded from the array, and folded back into the single friendly `apply-filter`
   action — keeping read symmetric with create.

6. **Correctness bar: round-trip stability, not input byte-identity.** A read result re-compiles to
   the same stored metadata (read → create → read is stable). Value formatting (float/datetime) may
   differ from the original human input; semantic/ESQ equivalence is the bar, guarded by round-trip
   property tests.

7. **Fail-closed on unmappable shapes.** If a stored rule cannot be faithfully reconstructed, the
   read fails with a clear, **rule-named** error rather than returning a raw or partial shape.

## Consequences

### Positive
- Lossless read for **every** entity rule, including UI-authored `apply-static-filter` rules — at no
  ongoing create-path cost and with no live queries.
- Read output drops straight into `create-entity-business-rule(s)` (read → edit → recreate).
- The reverse converter and ESQ decompiler are clean mirrors of the forward components and are
  fully unit-testable offline.

### Negative / trade-offs (explicitly accepted)
- **No identity in output (Decision 2/3).** Rules with duplicate captions are indistinguishable, and
  there is no handle for a future update/delete tool to target. Accepted for byte-identical symmetry;
  identity is deferred to the edit/delete work that needs it.
- **`enabled` not surfaced (Decision 2).** Disabled rules are returned in their create-contract form
  with no indication they are disabled (create always sets `enabled = true`).
- **ESQ format-drift coupling (Decision 4).** The decompiler mirrors the current wire format in
  `EsqEnvelopeDtos`. A platform format change would break forward and reverse together and is caught
  by the round-trip property tests.
- **Value fidelity (Decision 6).** Output is round-trip-stable but not guaranteed byte-identical to
  the first human input.
- **Formula name≠path edge.** Friendly input used the attribute *name*; stored metadata uses the
  *path*. Identical for direct fields; a documented edge when they differ.

## Alternatives considered
- **Persist friendly source at create (`{uId}.ClioFriendlySource`)** — the prior proposed ADR. Avoids
  a decompiler but modifies the create path and only helps clio-created rules going forward
  (UI-authored rules stay unreadable). Rejected in favor of the decompiler, which is lossless for all
  rules and leaves create untouched.
- **Fail-closed / read-only escape hatch for `apply-static-filter`** — simplest, no decompiler, but
  static-filter rules are unreadable or non-replayable. Rejected.
- **Output `name`/`enabled` identity (ENG-88580 shape)** — richer, but no in-scope consumer uses it.
  Rejected for this cut (Option 1); revisit with edit/delete.
- **list + get (two tools)** — ENG-88580 shape. Superseded by the single array tool.
