# How to Add a Lookup Quick Filter Menu Item (`crt.LookupQuickFilterMenuItem`) to a Freedom UI Page

> Audience: code agent inserting a `crt.LookupQuickFilterMenuItem` into a Creatio Freedom UI page schema.
>
> A `crt.LookupQuickFilterMenuItem` is a single checkbox row inside a lookup quick-filter
> dropdown. It is rarely authored by hand — the parent `crt.LookupQuickFilter` component
> generates these items through its preprocessor at runtime. Insert `crt.LookupQuickFilterMenuItem`
> directly only when building a fully custom quick-filter dropdown outside the standard filter
> pipeline.

## Metadata

- **Category**: filter
- **Container**: no
- **Parent types**: `crt.LookupQuickFilter` items slot (generated by preprocessor)
- **Typical children**: none

---

## 1. Mental model — the 1 place you must edit

| # | Section | What you add |
|---|---|---|
| 1 | `viewConfigDiff` | An `insert` op with `type: "crt.LookupQuickFilterMenuItem"` inside the parent filter container. |

> **Note:** In normal Freedom UI pages you add `crt.LookupQuickFilter` instead, and the platform
> preprocessor generates `crt.LookupQuickFilterMenuItem` items automatically. See the
> `crt.LookupQuickFilter` recipe for the canonical approach.

### 1.1 Naming convention

```
LookupQuickFilterMenuItem_<id>     // view element name
```

---

## 2. Step-by-step recipe

### 2.1 Insert the view element (`viewConfigDiff` entry)

```jsonc
{
  "operation": "insert",
  "name": "LookupQuickFilterMenuItem_abc",
  "parentName": "LookupFilterContainer",
  "propertyName": "items",
  "index": 0,
  "values": {
    "type": "crt.LookupQuickFilterMenuItem",
    "value": "some-guid",
    "displayValue": "Active",
    "checkedState": false,
    "checkedStateChange": { "request": "crt.LookupFilterItemCheckedStateChangeRequest" }
  }
}
```

---

## 3. Property reference

Full `inputs`, `outputs`, `default`, `values`, `deprecated`, and `designerDefaults` for
`crt.LookupQuickFilterMenuItem` are in `ComponentRegistry.json` under
`componentType: "crt.LookupQuickFilterMenuItem"`. This guide covers only the assembly mechanics.

## 5. Copy-paste minimal example

```jsonc
// viewConfigDiff entry (used inside a custom quick-filter container)
{
  "operation": "insert",
  "name": "StatusFilter_Active",
  "parentName": "StatusFilterContainer",
  "propertyName": "items",
  "index": 0,
  "values": {
    "type": "crt.LookupQuickFilterMenuItem",
    "value": "active-record-guid",
    "displayValue": "Active",
    "checkedState": false,
    "checkedStateChange": {
      "request": "crt.StatusFilterItemCheckedStateChangeRequest",
      "params": {
        "value": "active-record-guid"
      }
    }
  }
}
```

## 7. Common pitfalls

1. **Do not insert this component manually in standard filter workflows** — use `crt.LookupQuickFilter`
   and let the preprocessor generate items. Manual insertion bypasses the filter
   pipeline and breaks the connected datasource query.
2. **`checkedStateChange` is an input, not an output** — it is a callback function
   `(event: CheckedStateEvent) => void`, not a `RequestBindingConfig`; wire it to a
   function or omit it when the parent component manages state.
3. **`value` must match the lookup record Id (GUID string)** — mismatched values cause
   the filter to produce no results silently.
4. **No `@CrtInterfaceDesignerItem`** — this component cannot be placed via the
   interface designer toolbar; it is a sub-element only.

---

## 8. Quick checklist

- [ ] Consider using `crt.LookupQuickFilter` instead — it generates these items automatically.
- [ ] If inserting manually: `value` (GUID), `displayValue` (label text), and
  `checkedState` (initial toggle state) are all provided.
- [ ] `checkedStateChange` callback is wired when the parent needs to react to toggle events.
- [ ] Parent container is a valid lookup filter container or custom flex wrapper.
