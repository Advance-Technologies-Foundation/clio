# Research: MCP components and versioning (CDN model)

Research on the evolution of the Freedom UI component catalog that is exposed to AI through the clio MCP server.

## Context

The clio MCP server exposes to AI agents a set of tools for working with Creatio: creating pages, schemas, components, managing environments. One of the key tools — `get-component-info` — returns to AI a curated catalog of Freedom UI components with property descriptions.

Current baseline (committed in clio repo until this work lands):
- 92 components in 5 categories
- Single in-repo file [clio/Command/McpServer/Data/ComponentRegistry.json](../clio/Command/McpServer/Data/ComponentRegistry.json)
- No version metadata at all
- Loader reads via `IFileSystem` from the executing directory

This research defines the **target architecture**: the catalog is generated by CI in the `creatio-ui` monorepo on GA-tag, pushed to the `static-files-mcp` GitLab repository, mirrored to the academy.creatio.com CDN every 5 minutes, and consumed by clio over HTTPS with a layered fallback to a build-time-embedded snapshot.

## Architectural shift

A prior iteration (PR [clio#595](https://github.com/Advance-Technologies-Foundation/clio/pull/595), branch `research/mcp-components-versioning`) prototyped a three-domain NuGet-based delivery (`creatio-ui` → `creatio-component-registry-composer` → `clio`, with `Creatio.ComponentRegistry@0.1.0` published to nuget.org). After architectural review the team chose a **simpler two-domain CDN-based delivery**:

- The composer repo and the NuGet package are obsolete (composer-repo will be deleted; NuGet `Creatio.ComponentRegistry@0.1.0` will be unlisted and removed via nuget.org support).
- AI-side overrides (`aiHidden` / `aiOverlay`) are removed in v1 — `creatio-ui` CI emits the catalog directly.
- The fallback chain inside clio gains a runtime file cache between CDN and the embedded snapshot.

PR [clio#595](https://github.com/Advance-Technologies-Foundation/clio/pull/595) will be closed without merging; this branch (`research/mcp-components-cdn`) supersedes it.

## Key architectural decisions

- **Two-stage producer pipeline**: creatio-ui (source + CI; Jenkins job is planned and owned by the Platform-UI team) pushes generated `ComponentRegistry.json` files into the `static-files-mcp` GitLab repository (`gitdigital.creatio.com/academy/static-files-mcp`); the academy infrastructure mirrors that repository to the public CDN tree at `academy.creatio.com/api/mcp/` every 5 minutes. clio is the pure consumer side.
- **CDN as transport**: public HTTPS on academy.creatio.com, no auth. URL pattern `https://academy.creatio.com/api/mcp/{version}/ComponentRegistry.json` + `latest/ComponentRegistry.json` alias.
- **Per-version delivery**: each GA-tag (`8.2.0`, `8.2.1`, `8.3.0`, …) gets its own file. No per-entry `availability` ranges — the file is the unit of versioning.
- **JSON shape is drop-in compatible** with the current `ComponentRegistry.json` (top-level array of `ComponentRegistryEntry`). No wrapper, no `schemaVersion`, no `categories` (yet).
- **No AI-side overrides in v1.** `creatio-ui` CI emits the full extracted set. Curation is a future stage if needed.
- **Three-layer fallback in clio**: CDN → file cache (`~/.clio/cache/component-registry/`) → embedded snapshot in `clio.dll` (fetched at clio build time from CDN `latest.json` via MSBuild target; with a committed seed-snapshot for bootstrap and offline build).
- **Cache policy**: TTL 24h, **stale-while-revalidate** — expired cache is returned synchronously, refresh runs in the background. AI never blocks on the network.
- **Version resolution in clio**: internal stack `explicit target-version > GetSysInfo > latest`. In v1 the MCP tool surface activates only the lower two rungs (Args unchanged); `explicit` is reserved for a future tool-surface bump.
- **MCP contract**: `ComponentInfoArgs` unchanged. `ComponentInfoResponse` adds `resolvedTargetVersion` + `resolvedFrom` (`"environment"` | `"latest-fallback"`).
- **Trigger in creatio-ui CI**: GA-tag only. Branch-cut produces a Jenkins baseline artifact for regression detection (no git push to `static-files-mcp`, therefore no CDN refresh).

## Out of scope for Part 1 (this branch)

- Per-entry `Availability` ranges in the catalog. Versioning happens at the file level on the CDN.
- Explicit `target-version` parameter in `ComponentInfoArgs`. Internal resolver stack supports it; tool surface does not.
- Data-driven categories. The hardcoded `CategoryOrder` in clio remains.
- AI-side overrides (`aiHidden` / `aiOverlay`).
- Cross-version diff or compatibility filtering inside clio (each file is self-contained).
- Implementation of the Jenkins job inside the `creatio-ui` repo. This branch describes the contract in [jenkins-pipeline-spec.md](jenkins-pipeline-spec.md); the implementation is a separate cross-repo PR.

## Status

The documents describe the **target state**. Implementation in clio proceeds in the same branch after the architecture docs are reviewed. Producer-side coverage today: the `static-files-mcp` GitLab repository hosts the per-version files, and the academy infrastructure mirrors them to the CDN every 5 minutes — Jenkins automation in `creatio-ui` that generates the files and pushes to that repository is a parallel cross-repo track owned by the Platform-UI team; until it lands, the files are added to GitLab manually.

Delivery stages are in [implementation-plan-part1-cdn.md](implementation-plan-part1-cdn.md).
