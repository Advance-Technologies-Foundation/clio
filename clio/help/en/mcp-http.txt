COMMAND TYPE
    AI integration commands

NAME
    mcp-http - start MCP server in HTTP transport mode

DESCRIPTION
    Starts a Model Context Protocol (MCP) server that communicates over
    HTTP using the Streamable HTTP transport (JSON-RPC 2.0 over POST).
    This server exposes the same clio tools available in stdio mode to
    AI agents and platforms that connect over HTTP.

    The server runs until the process is terminated (Ctrl+C or SIGTERM).

PREREQUISITES
    mcp-http hosts the MCP endpoint on ASP.NET Core, so clio requires the
    ASP.NET Core shared runtime (Microsoft.AspNetCore.App) in addition to the
    base .NET runtime. The .NET SDK bundles it; runtime-only installs that ship
    only Microsoft.NETCore.App are not supported.

SECURITY
    The HTTP transport validates the Host header (restricted to the bound host)
    and the Origin header (restricted to the bound host / loopback) to prevent
    DNS-rebinding and cross-origin abuse; disallowed requests are rejected with
    HTTP 400 / 403. Native MCP clients send no Origin header and are unaffected.
    Standard OAuth 2.1 Resource-Server authorization is available (see
    --auth-authority below) but OFF by default. With it off, use --host 0.0.0.0
    only on trusted networks (a public/wildcard --host with authorization off
    refuses to start unless --allow-insecure-public is set).

OPTIONS
    --port <port>
        Port to listen on.
        Default: 8005

    --host <host>
        Host address to bind to.
        Default: 127.0.0.1

    --path <path>
        MCP endpoint path.
        Default: /mcp

    --platform-api-key <keys>
        Non-OAuth dev/offline fallback for per-request credential passthrough.
        Comma-separated set of platform API key(s) (also read from the
        CLIO_MCP_HTTP_PLATFORM_API_KEY environment variable). Setting at least
        one key ENABLES per-request credential passthrough; each request must
        then present a matching 'Authorization: Bearer <platform-api-key>'.
        Keys are compared in constant time; rotate by supplying the old and new
        key together in the comma-set. Unset (default) => passthrough disabled.
        IGNORED ENTIRELY when --auth-authority is configured -- standard OAuth
        is then the only front door and cannot be combined with this key on
        the same Authorization header. Configuring both logs a startup warning
        since the key becomes inert, not a security problem.

    --allowed-base-urls <urls>
        Comma-separated SSRF allowlist of origins a passthrough target url may
        reach. Each entry MUST include a scheme (e.g. https://tenant.creatio.com);
        bare hostnames fail at startup. When unset, only the always-on baseline
        blocks (cloud-metadata / link-local / loopback) apply and any other
        https host is allowed.

    --session-idle-ttl <duration>
        Idle time after which an unused per-session container is evicted.
        Accepts '90s' / '5m' / '1h' / '1d', bare seconds ('300'), or a TimeSpan
        ('00:05:00').
        Default: 5m

    --max-sessions <count>
        Maximum per-session containers kept in memory; the least-recently-used
        one is evicted when exceeded.
        Default: 50

    --credentials-header-name <name>
        Name of the request header carrying the base64-encoded JSON credentials.
        Default: X-Integration-Credentials

    --auth-authority <url>
        OIDC authority (discovery/JWKS base URL) of the OAuth 2.1 Authorization
        Server whose access tokens this edge accepts (also CLIO_MCP_HTTP_AUTH_AUTHORITY).
        Setting it ENABLES standard bearer-JWT authorization; unset => off (as before).

    --auth-audience <auds>
        Comma-separated accepted audience(s) validated against the token 'aud'
        (also CLIO_MCP_HTTP_AUTH_AUDIENCE).

    --auth-required-scopes <scopes>
        Comma-separated scope(s) every request must carry, all required, checked
        against the 'scope'/'scp' claim (also CLIO_MCP_HTTP_AUTH_REQUIRED_SCOPES).

    --auth-issuer <issuers>
        Comma-separated accepted issuer(s) for the token 'iss' (also
        CLIO_MCP_HTTP_AUTH_ISSUER). Optional; when unset the issuer is validated
        against the discovery document. Use when the public token 'iss' differs
        from the internal --auth-authority discovery URL.

    --auth-allow-insecure-metadata
        Allow OIDC metadata/JWKS over plain HTTP (also a truthy
        CLIO_MCP_HTTP_AUTH_ALLOW_INSECURE_METADATA). Default is HTTPS-only; use
        only for an internal-DNS HTTP authority on a trusted network.

    --auth-resource <uri>
        Explicit Protected Resource Metadata 'resource' (canonical MCP endpoint
        URI) override (also CLIO_MCP_HTTP_AUTH_RESOURCE). Unset (default) =>
        derived per-request from the incoming scheme/host/path, correct behind
        any ingress that forwards the Host header.

    --allow-insecure-public
        Allow starting with a public/wildcard --host (e.g. 0.0.0.0) while
        authorization is OFF (also a truthy
        CLIO_MCP_HTTP_ALLOW_INSECURE_PUBLIC). Without this flag, clio REFUSES TO
        START in that combination: an unauthenticated public bind exposes every
        registered environment's stored credentials to anyone who can reach the
        port. Not recommended outside a fully trusted network.

    When authorization is enabled, EVERY request to the MCP endpoint requires a
    valid token — pre-registered '-e <env>' / stored-credential access included,
    not just the credential-passthrough leg. The edge serves Protected Resource
    Metadata (RFC 9728) anonymously at /.well-known/oauth-protected-resource, and
    an unauthenticated/invalid-token/insufficient-scope request receives 401 (or
    403 for scope) with a WWW-Authenticate header naming that discovery URL.

    With no --auth-authority configured (default), authorization is off and the
    server behaves exactly as before this feature — EXCEPT that a public/wildcard
    --host with no authorization refuses to start (see --allow-insecure-public).

CREDENTIAL PASSTHROUGH (multi-tenant edge)
    Per-request credential passthrough lets a gateway target a different Creatio
    tenant on each request instead of a pre-registered environment. When
    --auth-authority is NOT configured (default), it is gated SOLELY by the
    platform API key and off by default (fail-closed):

      - Set a platform API key (--platform-api-key / CLIO_MCP_HTTP_PLATFORM_API_KEY)
        to enable it.

    With no key configured, the credential header is ignored and the server
    behaves exactly as before — the verb, stdio mode, and '-e <env>' targeting
    are always available regardless of the key.

    When --auth-authority IS configured, the platform API key is IGNORED
    entirely (see --platform-api-key above) and passthrough eligibility comes
    solely from the authenticated OAuth principal instead.

    Standard-auth header strip: when --auth-authority IS configured, the
    credential header is honored ONLY on an authenticated request (a valid
    bearer token) -- an unauthenticated request has it ignored outright, not
    deferred. The inbound gateway/platform bearer token is never forwarded to
    Creatio; the outbound credential comes solely from this header. Per-tenant
    claim-based authorization (which gateway may assert which tenant) is not
    implemented in v1 -- the platform's client-credentials token carries no
    per-tenant claim yet; any authenticated caller is trusted for any tenant,
    same trust boundary as the platform-api-key gate had.

    When enabled, an authorized request carries credentials in the header:

        Authorization: Bearer <platform-api-key>
        X-Integration-Credentials: <base64 JSON>

    The base64 JSON payload requires 'url' plus auth material, resolved by
    precedence accessToken -> login+password:

        {"url":"https://tenant.creatio.com","accessToken":"<access-token>"}
        {"url":"https://tenant.creatio.com","login":"<login>","password":"<password>"}

    An access token must be Bearer-typed. A cookie-only payload parses but is
    REJECTED as unsupported in v1 — supply an access token instead.

    Nothing is persisted: clio builds an ephemeral, in-memory container per
    tenant (never written to settings/disk), pooled with idle-TTL / LRU
    eviction (--session-idle-ttl / --max-sessions).

    While passthrough is active for a request, supplying uri/login/password/
    client-id/client-secret or an environment name as tool ARGS is rejected —
    credentials must ride the header, not the arguments.

    Residual: a hostname that passes --allowed-base-urls but resolves to a
    blocked IP AFTER validation (DNS-rebinding TOCTOU) is out of scope for v1.

EXAMPLES
    Start with default settings (port 8005, localhost):
        clio mcp-http

    Start on a custom port:
        clio mcp-http --port 9000

    Listen on all network interfaces (no authentication — use only on trusted networks):
        clio mcp-http --host 0.0.0.0

    Custom endpoint path:
        clio mcp-http --path /api/mcp

    Enable the multi-tenant credential-passthrough edge (a platform API key turns it on):
        clio mcp-http --host 0.0.0.0 \
            --platform-api-key <platform-api-key> \
            --allowed-base-urls https://tenant.creatio.com

ENVIRONMENT VARIABLES
    CLIO_MCP_RESPECT_AMBIENT_PROXY
        When set to "true" or "1", the HTTP MCP server does NOT neutralize
        ambient proxy environment variables. Default behavior (absent or
        any other value) bypasses all inherited proxies.

    CLIO_MCP_HTTP_PLATFORM_API_KEY
        Comma-separated platform API key(s), unioned with --platform-api-key.
        Setting at least one key enables per-request credential passthrough.
